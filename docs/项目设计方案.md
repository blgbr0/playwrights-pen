# PlaywrightsPen 项目设计方案

> 剧作家之笔 - 自然语言驱动的自动化测试服务

## 1. 项目概述

### 1.1 项目定位

PlaywrightsPen 是一个将**自然语言测试描述**转换为**可执行浏览器自动化测试**的智能测试服务。它利用 LLM（大语言模型）解析用户的自然语言输入，并通过 Playwright MCP 协议控制浏览器执行测试。

### 1.2 核心价值

| 特性 | 传统自动化测试 | PlaywrightsPen |
|------|---------------|----------------|
| 学习成本 | 需要学习编程和测试框架 | 使用自然语言描述即可 |
| 维护成本 | 元素定位符变化需手动更新 | LLM 智能识别元素 |
| 可读性 | 代码可读性依赖开发者 | 自然语言天然可读 |
| 执行模式 | 单一执行模式 | 支持探索/回归双模式 |

## 3. 数据模型

### 3.1 核心实体关系

```mermaid
erDiagram
    TestCase ||--o{ TestStep : contains
    TestCase ||--o{ Session : executes
    Session ||--o{ StepExecution : records

    TestCase {
        string id PK "12位UUID"
        string name "用例名称"
        string description "自然语言描述"
        list steps "解析后的步骤"
        list tags "分类标签"
        datetime created_at "创建时间"
        datetime updated_at "更新时间"
        string last_session_id FK "最近执行会话"
    }

    TestStep {
        ActionType action "动作类型"
        string url "导航URL"
        string text "输入文本"
        string selector_hint "元素描述"
        AssertionType assertion_type "断言类型"
        string expected_value "期望值"
        string recorded_ref "录制的元素引用"
        bool is_key_step "是否关键步骤"
        string description "步骤描述"
    }

    Session {
        string id PK "会话ID"
        string test_case_id FK "关联用例"
        ExecutionMode mode "执行模式"
        SessionStatus status "会话状态"
        int current_step_index "当前步骤"
        list step_executions "执行记录"
        datetime started_at "开始时间"
        datetime ended_at "结束时间"
        int total_steps "总步骤数"
        int passed_steps "通过数"
        int failed_steps "失败数"
    }

    StepExecution {
        int step_index "步骤索引"
        SessionStatus status "执行状态"
        datetime started_at "开始时间"
        datetime ended_at "结束时间"
        string snapshot_before "执行前快照"
        string snapshot_after "执行后快照"
        string llm_reasoning "LLM推理过程"
        string element_ref_used "使用的元素引用"
        string result "执行结果"
        string error "错误信息"
        bool user_confirmed "用户已确认"
    }
```

### 3.2 枚举类型定义

#### ActionType - 动作类型

| 值 | 说明 | 必需参数 |
|---|------|---------|
| `navigate` | 导航到 URL | `url` |
| `click` | 点击元素 | `selector_hint` |
| `type` | 输入文本 | `selector_hint`, `text` |
| `select` | 选择下拉选项 | `selector_hint`, `text/values` |
| `hover` | 鼠标悬停 | `selector_hint` |
| `scroll` | 页面滚动 | `direction`, `amount` |
| `wait` | 等待 | `time_ms` |
| `screenshot` | 截图 | `filename` (可选) |
| `assert` | 断言验证 | `assertion_type`, `expected_value` |
| `execute_js` | 执行 JavaScript | `function` |

#### AssertionType - 断言类型

| 值 | 说明 | 示例 |
|---|------|------|
| `text_contains` | 页面/元素包含文本 | 验证页面包含"登录成功" |
| `text_equals` | 文本完全匹配 | 验证标题等于"首页" |
| `element_visible` | 元素可见 | 验证"提交按钮"可见 |
| `element_exists` | 元素存在 | 验证"错误提示"存在 |
| `url_contains` | URL 包含 | 验证 URL 包含"/dashboard" |
| `url_equals` | URL 完全匹配 | 验证 URL 等于"https://..." |
| `title_contains` | 标题包含 | 验证标题包含"管理后台" |
| `title_equals` | 标题完全匹配 | - |

#### ExecutionMode - 执行模式

| 值 | 说明 |
|---|------|
| `exploration` | 探索模式：首次执行，LLM 实时定位元素 |
| `regression` | 回归模式：使用已录制的元素引用快速执行 |
| `hybrid` | 混合模式：优先使用录制，失败时回退 LLM |

#### SessionStatus - 会话状态

| 值 | 说明 |
|---|------|
| `pending` | 等待开始 |
| `running` | 执行中 |
| `paused` | 暂停等待确认 |
| `passed` | 测试通过 |
| `failed` | 测试失败 |
| `aborted` | 用户中止 |

#### ConfirmationMode - 确认模式

| 值 | 说明 |
|---|------|
| `every_step` | 每步都需确认 |
| `key_steps` | 仅关键步骤需确认 |
| `none` | 完全自动执行 |

## 4. 业务流程

### 4.1 测试用例生命周期

```mermaid
stateDiagram-v2
    [*] --> Created: 创建用例
    Created --> Parsed: LLM 解析
    Parsed --> Exploration: 首次执行
    Exploration --> Recorded: 执行成功
    Exploration --> Failed: 执行失败
    Recorded --> Regression: 回归测试
    Regression --> Passed: 回归通过
    Regression --> Failed: 回归失败
    Failed --> Parsed: 重新解析
    Passed --> [*]
```

### 4.2 探索模式执行流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant CLI as CLI/API
    participant Orch as Orchestrator
    participant Parser as TestParser
    participant LLM as LLMClient
    participant Exec as Executor
    participant MCP as MCPClient
    participant Browser as 浏览器

    User->>CLI: 输入自然语言描述
    CLI->>Orch: run_exploration(testcase)
    
    alt 未解析
        Orch->>Parser: parse(description)
        Parser->>LLM: 调用 LLM 解析
        LLM-->>Parser: 返回步骤 JSON
        Parser-->>Orch: 返回 TestStep 列表
    end

    Orch->>MCP: connect()
    MCP->>Browser: 启动浏览器

    loop 每个步骤
        Orch->>Exec: execute_step(step)
        Exec->>MCP: get_snapshot()
        MCP->>Browser: 获取可访问性树
        Browser-->>MCP: 返回快照
        MCP-->>Exec: 返回快照文本
        
        alt 需要定位元素
            Exec->>LLM: locate_element(snapshot, hint)
            LLM-->>Exec: 返回元素引用
        end
        
        Exec->>MCP: 执行动作 (click/type/...)
        MCP->>Browser: 执行浏览器操作
        Browser-->>MCP: 操作结果
        MCP-->>Exec: 返回结果
        Exec-->>Orch: 返回 StepExecution
        
        alt 关键步骤
            Orch->>User: 请求确认
            User-->>Orch: 确认/拒绝
        end
    end

    Orch->>MCP: close()
    MCP->>Browser: 关闭浏览器
    Orch-->>CLI: 返回 Session
    CLI-->>User: 显示结果
```

### 4.3 回归模式执行流程

```mermaid
sequenceDiagram
    participant Orch as Orchestrator
    participant Exec as Executor
    participant MCP as MCPClient
    participant LLM as LLMClient

    Note over Orch,LLM: 回归模式使用已录制的元素引用

    loop 每个步骤
        Orch->>Exec: execute_step(step, use_recorded_ref=True)
        
        alt 有录制引用
            Exec->>MCP: 直接使用 recorded_ref
            Note right of Exec: 跳过 LLM 调用，速度更快
        else 无录制引用
            Exec->>LLM: locate_element() 回退到 LLM
        end
        
        MCP-->>Exec: 执行结果
        Exec-->>Orch: StepExecution
    end
```

### 4.4 混合模式执行流程 (Script + 自然语言)

混合模式允许在回放已录制的脚本时，通过自然语言灵活处理数据变化或流程变更。

#### 使用场景

| 场景 | 传统方式 | 混合模式 |
|------|----------|----------|
| 测试数据变化 | 重新录制整个脚本 | 仅修改占位符值 |
| 业务流程调整 | 重写测试用例 | 插入/替换自然语言步骤 |
| 元素定位失败 | 手动修复选择器 | 自动切换 LLM 智能定位 |
| 新增步骤 | 修改代码 | 用自然语言描述新步骤 |

#### 脚本语法扩展

```yaml
# 原始录制脚本格式
steps:
  - action: navigate
    url: "https://example.com/login"
    recorded_ref: "page1"
  
  - action: type
    selector_hint: "用户名输入框"
    text: "{{username}}"           # 使用占位符
    recorded_ref: "input[name=user]"
  
  - action: click
    selector_hint: "登录按钮"
    recorded_ref: "button[type=submit]"
  
  # ✨ 混合模式：插入自然语言步骤
  - action: natural_language
    description: "如果出现验证码弹窗，点击发送验证码并等待3秒"
    
  # ✨ 混合模式：条件分支
  - action: conditional
    condition: "如果页面显示'新用户引导'"
    then_steps:
      - action: natural_language
        description: "完成新用户引导流程，跳过所有可选步骤"
    else_steps:
      - action: natural_language
        description: "继续正常流程"
```

#### 执行流程

```mermaid
sequenceDiagram
    participant Orch as Orchestrator
    participant Exec as Executor
    participant LLM as LLMClient
    participant MCP as MCPClient

    loop 每个步骤
        Orch->>Exec: execute_step(step)
        
        alt 普通步骤 (有 recorded_ref)
            Exec->>MCP: 尝试使用录制引用
            alt 引用有效
                MCP-->>Exec: 执行成功
            else 引用失效
                Exec->>LLM: 转为自然语言定位
                LLM-->>Exec: 返回新元素引用
                Note right of Exec: 更新 recorded_ref
            end
        else 自然语言步骤 (action=natural_language)
            Exec->>LLM: 解析自然语言描述
            LLM-->>Exec: 返回子步骤
            loop 子步骤
                Exec->>MCP: 执行子步骤
            end
        else 条件步骤 (action=conditional)
            Exec->>MCP: 获取页面快照
            Exec->>LLM: 评估条件是否满足
            LLM-->>Exec: true/false
            alt 条件满足
                Exec->>Orch: 执行 then_steps
            else 条件不满足
                Exec->>Orch: 执行 else_steps
            end
        end
    end
```

#### 数据占位符覆盖

运行时可通过参数覆盖脚本中的占位符值：

```bash
# CLI 方式
playwrights-pen run-script testcase.yaml \
  --set username="newuser@test.com" \
  --set password="NewP@ssw0rd"

# API 方式
POST /api/sessions
{
  "test_case_id": "abc123",
  "mode": "hybrid",
  "data_overrides": {
    "username": "newuser@test.com",
    "password": "NewP@ssw0rd"
  }
}
```


### 4.4 元素定位流程

```mermaid
flowchart TD
    A[开始定位元素] --> B{是否有 recorded_ref?}
    B -->|是且回归模式| C[直接使用 recorded_ref]
    B -->|否或探索模式| D[获取页面快照]
    
    D --> E[调用 LLM locate_element]
    E --> F{LLM 返回成功?}
    
    F -->|是| G[保存 ref 到 step.recorded_ref]
    G --> H[返回元素引用]
    
    F -->|否| I[抛出定位失败错误]
    
    C --> J{元素仍然有效?}
    J -->|是| H
    J -->|否| D
```

## 5. API 接口设计

### 5.1 测试用例 API

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/testcases` | 创建测试用例 |
| `GET` | `/api/testcases` | 列出所有用例 |

| `PUT` | `/api/testcases/{id}` | 更新用例 |
| `DELETE` | `/api/testcases/{id}` | 删除用例 |

#### 创建用例请求

```json
{
  "name": "百度搜索测试",
  "description": "打开百度，搜索 Playwright，验证搜索结果包含 Playwright 关键词",

  "parse_now": true
}
```

#### 用例响应

```json
{
  "id": "abc123def456",
  "name": "百度搜索测试",
  "description": "...",
  "tags": ["搜索", "冒烟测试"],
  "step_count": 4,
  "has_steps": true
}
```

### 5.2 执行会话 API

| 方法 | 路径 | 说明 |

## 8. LLM 提示词设计

### 8.1 测试用例解析提示词

```
你是一个测试用例解析器。将用户的自然语言测试描述转换为结构化的测试步骤。

输出 JSON 格式，包含 steps 数组，每个步骤包含:
- action: 动作类型
- url: 导航 URL
- text: 输入文本
- selector_hint: 元素描述
- assertion_type: 断言类型
- expected_value: 期望值
- is_key_step: 是否关键步骤
- description: 步骤描述
```

### 8.2 元素定位提示词

```
你是一个 Web 元素定位专家。根据可访问性树快照和用户描述，找到对应的元素引用。

可访问性树格式:
- heading "标题" [ref=e1]
- textbox "搜索" [ref=e2]
- button "提交" [ref=e3]

返回 JSON: {"ref": "e2", "reasoning": "根据描述匹配到搜索框"}
```

## 9. 扩展性设计

### 9.1 当前架构优势

1. **模块化设计**: 各组件职责单一，易于替换
2. **配置驱动**: 关键参数通过环境变量配置
3. **LLM 无关性**: 支持任何 OpenAI 兼容的 LLM 服务
4. **存储抽象**: Repository 模式便于切换存储后端

### 9.2 未来扩展方向

| 方向 | 描述 |
|------|------|
| 数据库存储 | 替换文件存储为 SQLite/PostgreSQL |
| Web 管理界面 | 提供可视化的测试管理和执行界面 |
| 并行执行 | 支持多浏览器并行执行测试 |
| 报告生成 | 生成 HTML/PDF 格式的测试报告 |
| CI/CD 集成 | 提供 GitHub Actions / Jenkins 插件 |
| 多浏览器支持 | 扩展支持 Firefox, WebKit |
